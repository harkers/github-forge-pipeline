# Security Implementation Guide

## Security objective

Keep durable memory useful without turning the OpenClaw server into a context leak machine.

## Threat-aware design choices

### 1. Workspace-local durable memory by default
Durable knowledge belongs under the target workspace by default:
- `para/`
- `memory/`
- `MEMORY.md`
- `AGENTS.md`

This reduces:
- cross-project memory bleed
- accidental broad indexing
- unclear ownership of facts
- retrieval of unrelated sensitive context

### 2. Files are canonical truth
Semantic search layers are useful, but they are not canonical truth.
The source of truth is:
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `para/`

### 3. Tiny routing layer
`MEMORY.md` should stay small.
Do not use it as a dumping ground for:
- raw transcripts
- secrets
- credentials
- broad personal context
- tool outputs

### 4. Managed-block patching
Managed blocks make the installer:
- idempotent
- reviewable
- safer to re-run
- less destructive than rewriting the whole file

### 5. Path safety
The installer:
- canonicalizes the target workspace path
- rejects symlinked target files
- restricts `--para-root` to a relative path unless the operator intentionally changes the script
- keeps writes inside the workspace

### 6. Permission posture
The scripts use:
- `umask 077`
- directory checks
- file mode verification

Recommended target posture:
- workspace private to the OpenClaw OS user
- sensitive files at `0600`
- directories at `0700`

### 7. Hidden config is review-first
This pack does not edit `~/.openclaw/openclaw.json`.
That is deliberate.
Production config deserves:
- review
- diff visibility
- rollback planning
- gateway restart coordination

### 8. Secret handling
Do not store:
- API keys
- bearer tokens
- cookies
- connection strings
- SSH private material

If you need secrets in OpenClaw, use the gateway's secret management features rather than plain text in memory files.

## Operational guardrails

- Run `openclaw security audit --deep` after rollout.
- Keep one trusted operator boundary per gateway.
- If you need adversarial-user isolation, split gateways or hosts.
- Treat the workspace as memory and keep it private.
- Prefer the smallest access surface that still works.

## Recommended production profile

### Good
- single trusted operator boundary
- one workspace per major trust domain
- workspace-local durable memory
- file-first memory
- optional QMD on a trial basis only
- direct verification after every installer run

### Bad
- shared global durable memory for unrelated projects
- direct hidden-config mutation from copied snippets
- enabling QMD and assuming the gateway bridge is correct without testing
- indexing the operator's whole home directory
- storing secrets in memory files

# QMD Integration Pattern

## Bottom line

Use QMD as an optional retrieval accelerator.
Do not make it canonical truth.
Do not make production success depend on OpenClaw's native QMD bridge alone.

## Why this pattern

The stable part of OpenClaw memory is the file layer.
QMD can add value by indexing those files and improving retrieval, but the safer production pattern is:

1. files remain canonical
2. QMD indexes only approved workspace paths
3. direct QMD query behaviour is tested first
4. agent-facing QMD usage is promoted later, after evidence

## Recommended scope

Index only:
- `<workspace>/MEMORY.md`
- `<workspace>/memory/`
- `<workspace>/para/`

Do not index:
- `~`
- unrelated workspaces
- config/secret directories
- browser/profile/cache locations

## Promotion model

### Phase 1 — file layer only
Use the pack in `core` or `core-plus-guidance`.
No QMD dependency.

### Phase 2 — sidecar validation
Install and test QMD separately.
Confirm:
- index builds
- re-index works
- direct query returns expected results
- failure behaviour is acceptable

### Phase 3 — controlled agent usage
Only after successful validation:
- expose a constrained QMD path to the operator
- compare results with direct file reads
- keep files canonical

### Phase 4 — optional native bridge testing
If you want to test OpenClaw's native QMD backend:
- do it in a lower-risk environment first
- verify retrieval and fallback behaviour
- keep a rollback path

## Safety principles

- separate indexing from truth
- separate testing from promotion
- separate retrieval convenience from durable governance

## Example operator questions before promotion

- Does direct QMD query return expected snippets from `MEMORY.md` and `para/`?
- Does re-index succeed without duplicate-path failures?
- Does the gateway actually return QMD results or silently fall back?
- Are all indexed paths inside the intended workspace?
- Can I disable QMD and still retain full durable memory in files?

## Practical recommendation

Today, the safest production posture is:
- keep file memory primary
- treat QMD as optional
- do not hinge production memory correctness on QMD alone
